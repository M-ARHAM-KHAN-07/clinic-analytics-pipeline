03 — Incremental Sync Layer

Watermark-based upserts

Custom Python layer that runs on top of Fivetran-synced data — moves rows incrementally from source schemas into the CRM warehouse with conflict resolution.

Each table has its own watermark column and composite primary key. The sync reads the current MAX(watermark) from the target, then only fetches rows newer than that value — avoiding full table scans on every run.

Google Ads — account_stats
Conflict key: customer_id, date, _fivetran_id
Watermark: date
Facebook Ads — basic_ad
Conflict key: ad_id, date, _fivetran_id
Watermark: date
incremental_ads_to_crm.py — watermark upsert 
# 1. Read current high-water mark from target table
last_wm = get_max_watermark(cur, target_schema, target_table, wm_col)

# 2. Only fetch rows newer than the watermark
select_query = f"SELECT {col_list} FROM {source_schema}.{source_table} s"
if last_wm:
    select_query += f" WHERE s.{wm_col} > %s"
    params.append(last_wm)

# 3. Upsert with per-table conflict key
final_query = f"""
    INSERT INTO {target_schema}.{target_table} ({col_list})
    {select_query}
    ON CONFLICT ({conflict_cols_sql})
    DO NOTHING;
"""

cur.execute(final_query, params)
logger.info("Rows inserted: %d", cur.rowcount)
04 — Data Quality

Clinic name mapping layer

The same clinic appears under different names across Google Ads, Facebook Ads, and internal systems. Built a shared mapping table that normalises names at load time across all pipelines.

The problem across 50+ clinics

A single clinic might appear as "67th Street Dental" in Google Ads, "67th St Dental - NYC" on Facebook, and "67 Street Dental Care" in the internal EDMS system. Without normalisation, dashboards show them as three separate clinics.

SELECT
    COALESCE(m.new_name, s.clinic_name) AS clinic,
    s.metric_1,
    s.metric_2
FROM source_table s
LEFT JOIN crm_data.fb_google_clinics_mapping m
    ON m.old_name = s.clinic_name;

07 — Engineering notes

Problems solved

01
Same clinic, five different names
Google Ads, Facebook Ads, QuickBooks, the EDMS portal, and the CSV target files all used slightly different names for the same physical clinic. Without normalisation, dashboards would fragment each clinic into multiple entries. Built a shared fb_google_clinics_mapping table and applied it via LEFT JOIN + COALESCE at load time in every pipeline that touches clinic names.
02
Full re-load vs incremental — each table is different
Different tables have different primary keys and watermark columns. Account history uses (id, _fivetran_start), stats tables use (customer_id, date, _fivetran_id). Each table entry in the config carries its own conflict_cols and watermark_col, so the generic upsert function handles all cases correctly without hardcoding.
03
CSV column headers were inconsistent across clinics
Target/goal CSVs uploaded to Azure Blob used different column names per clinic — "annual", "annual_target", "year_target" all meant the same thing. Built a find_col(possible_names) function that searches the header row for any matching alias, making the loader tolerant of naming variations without manual preprocessing.
04
EDMS portal has no API — modal-based date picker
SecureEDMS exposes reports only through a web portal with a JavaScript modal containing date inputs. Standard Selenium sends weren't registering because the inputs needed both DOM value injection and JS event dispatch. Solved with a js_set() helper that fires input, change, and blur events after setting the value, plus iframe context switching for the modal.
